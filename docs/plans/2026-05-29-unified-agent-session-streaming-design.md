# Unified Agent Session Streaming Design

Date: 2026-05-29

## Goal

Introduce a shared session and message foundation for all agent conversations
and graph-backed agent workflows. SOP quality checks should be the first
consumer, and the same foundation should later support agent management UI
flows where users create, test, and debug sub-agents through multi-turn
conversation.

The design separates durable transcript storage from LangGraph checkpoints and
from live token streaming:

- `sessions` and `messages` store the product-facing conversation transcript.
- LangGraph checkpoint tables store execution recovery state.
- Live token deltas are broadcast online only and are not persisted.
- Business tables link to `sessions.id` instead of each business domain owning a
  separate event log.

## Non-Goals

- Do not persist every token delta.
- Do not expose raw LangGraph checkpoint state to the frontend.
- Do not make SOP quality checks a plain chat flow. SOP quality remains a graph;
  DeepAgent is only one node inside that graph.
- Do not make graph nodes depend on SSE, EventSource, or database repositories
  directly.
- Do not design a wide generic business table that contains SOP fields, agent
  test fields, and future domain fields.

## Current State

The current PR already moved SOP quality checks away from the old generic
`runs/run_events` path and introduced:

- `sop_quality_checks`: SOP check business state, active uniqueness, result, and
  checkpoint pointer.
- `sop_quality_events`: lightweight SOP lifecycle events for replay and SSE.
- LangGraph Postgres checkpoints for graph execution state.
- A SOP graph with `load_sop -> review_sop -> summarize_result -> submit_result`.
- A provider-backed `AgentFactory` that creates fresh DeepAgent instances.

The problem is that `review_sop` currently consumes `agent.astream()` directly.
That makes a graph node responsible for runtime streaming, reasoning detection,
delta aggregation, and frontend-oriented live events. The same logic will also
be needed by future agent playground sessions, so it should move into a shared
runtime layer.

## Target End State

All agent-facing conversations and graph-backed agent workflows use two shared
tables:

```text
sessions
messages
```

Business domains associate with this foundation by holding a `session_id`.

Examples:

```text
sop_quality_checks.session_id -> sessions.id
agent_test_sessions.session_id -> sessions.id
future_domain_runs.session_id -> sessions.id
```

LangGraph uses `sessions.thread_id` as its `configurable.thread_id`. The
database primary key `sessions.id` is internal and stable for foreign keys.

The final architecture is:

```text
Business domain
  - sop_quality_checks
  - future agent_test_sessions
  - future domain-specific run records

Shared transcript
  - sessions
  - messages

Shared runtime
  - SessionMessageWriter
  - DeepAgentStreamRunner
  - future LangChainAgentStreamRunner
  - SessionStreamService

Execution recovery
  - LangGraph Postgres checkpoints

Transport
  - /api/sessions/{id}
  - /api/sessions/{id}/messages
  - /api/sessions/{id}/stream
  - domain-specific compatibility endpoints where useful
```

## Data Model

### `sessions`

`sessions` is a generic runtime conversation/session row. It has no `user_id`
for now. Authorization and ownership can be added later through business tables
or dedicated ACL tables if needed.

```text
id bigint generated by default as identity primary key
thread_id text not null unique
status text not null
title text null
created_at timestamptz not null default now()
updated_at timestamptz not null default now()
```

Recommended status values:

```text
active
completed
failed
interrupted
```

`thread_id` is the durable runtime identifier passed to LangGraph:

```python
{"configurable": {"thread_id": session.thread_id, "checkpoint_ns": ""}}
```

### `messages`

`messages` is the durable product-facing transcript. It stores complete
messages, not token deltas.

```text
id uuid primary key
session_id bigint not null references sessions(id)
sequence bigint not null
role text not null
content text not null
additional_kwargs jsonb not null default '{}'::jsonb
created_at timestamptz not null default now()
```

Recommended constraints and indexes:

```sql
create unique index uq_messages_session_sequence
on messages (session_id, sequence);

create index ix_messages_session_created_at
on messages (session_id, created_at);
```

`sequence` is session-local and is the only durable SSE replay cursor. UUID
message IDs are stable object identifiers, but clients should not use them as
the stream cursor.

Recommended roles:

```text
user
assistant
tool
system
```

`content` stores user-visible text or markdown. Tool UI metadata, graph step
metadata, and runtime references live in `additional_kwargs`.

Example `additional_kwargs`:

```json
{
  "kind": "final_message",
  "step": "review_sop",
  "turn_id": "0eb1...",
  "channel": "content",
  "checkpoint_id": "1f...",
  "model": "deepseek-v4-pro"
}
```

Tool call example:

```json
{
  "kind": "tool_call",
  "step": "submit_result",
  "tool_call_id": "call_123",
  "tool_name": "submit_quality_result",
  "status": "running",
  "args": {
    "check_id": "..."
  }
}
```

Tool result example:

```json
{
  "kind": "tool_result",
  "step": "submit_result",
  "tool_call_id": "call_123",
  "tool_name": "submit_quality_result",
  "status": "completed"
}
```

### SOP Business Link

`sop_quality_checks` remains the SOP business table. It should gain:

```text
session_id bigint not null references sessions(id)
```

Its existing `thread_id` can remain during migration, but the target state is
that `thread_id` is read from `sessions.thread_id`. `sop_quality_checks` keeps
SOP-specific fields:

```text
sop_id
env_key
graph_name
graph_version
status
quality_result
sop_snapshot
result
error
current_checkpoint_id
created_at / updated_at / started_at / finished_at
```

The active uniqueness rule remains on `sop_quality_checks`:

```sql
create unique index uq_sop_quality_checks_active_subject_env
on sop_quality_checks (sop_id, env_key)
where status in ('pending', 'running');
```

### Future Agent Test Link

Agent CRUD tables remain configuration and publication tables:

```text
agents
agent_versions
```

They should not store test conversation history.

Future agent playground sessions can use a thin business table:

```text
agent_test_sessions
- id uuid primary key
- session_id bigint not null references sessions(id)
- agent_id uuid not null references agents(id)
- agent_version_id uuid null references agent_versions(id)
- status text not null
- created_at / updated_at / finished_at
```

The actual multi-turn transcript remains in `messages`.

### `sop_quality_events` Sunset

`sop_quality_events` is a transitional SOP-specific event log. It should not be
the long-term generic event foundation.

First landing can keep it for compatibility with current SOP endpoints and old
checks. New runtime-facing code should write durable transcript data to
`messages`. Over time:

1. SOP detail returns `session_id`.
2. SOP stream can proxy or map from the generic session stream.
3. Frontend SOP UI reads from session messages and projects by
   `additional_kwargs.step`.
4. Once compatibility is no longer needed, `sop_quality_events` can be removed
   or limited to legacy read paths.

## Checkpoints Versus Messages

LangGraph checkpoints and `messages` have different jobs.

LangGraph checkpoint tables store machine recovery state:

- graph state values
- next node and task state
- model and tool message history needed for execution
- checkpoint IDs
- thread and namespace state

`messages` stores product-facing transcript state:

- what the user said
- what assistant response should be shown after refresh
- tool call/result cards for UI
- graph step output that should be visible to users
- stable session-local replay cursor

The frontend should not read checkpoint state directly. Checkpoints remain the
execution recovery authority. `sessions/messages` are the UI and transcript
authority.

## Runtime Components

### `SessionMessageWriter`

Graph nodes and domain services depend on a narrow writer protocol, not on
repositories or SSE:

```python
class SessionMessageWriter(Protocol):
    async def append_step_message(
        self,
        *,
        step: str,
        role: str,
        content: str,
        additional_kwargs: dict[str, Any] | None = None,
    ) -> SessionMessage:
        ...
```

The implementation handles:

- session lookup
- session-local sequence allocation
- message insert
- commit policy
- persisted message broadcast

Nodes should only append user-visible step transcript. They should not append
technical lifecycle events such as "node started", "node completed", retries, or
checkpoint internals.

### `DeepAgentStreamRunner`

SOP `review_sop` should explicitly use a DeepAgent-specific runner. Naming
should make the dependency clear:

```python
result = await deepagent_stream_runner.run_step(
    session=session,
    step="review_sop",
    input=DeepAgentRunInput(messages=[...]),
)
return {"review_output": result.final_text}
```

The runner should prefer streaming:

```text
if agent has astream:
  use astream(stream_mode=["messages", "updates", "custom"])
else if agent has ainvoke:
  use ainvoke
else:
  use invoke
```

Responsibilities:

- call `agent.astream(...)`
- extract content deltas
- detect `reasoning_content` and emit only a "thinking started" live signal
- never persist or expose reasoning content
- aggregate deltas into `final_text`
- write the final assistant message to `messages`
- broadcast live token deltas without persisting them
- return `DeepAgentRunResult(final_text=...)` to the graph node

`DeepAgentRunInput.metadata` can be used for logging and event metadata, but it
should not be injected into the model payload. The model payload stays close to
the LangChain shape:

```python
{"messages": input.messages}
```

### Ordinary Graph Nodes

SOP quality is a graph, not a single chat conversation. The session represents
one graph run; `messages.additional_kwargs.step` identifies graph node output.

Example transcript items:

```text
step=load_sop, role=tool/system, content="SOP snapshot loaded."
step=review_sop, role=assistant, content=<DeepAgent final review>
step=summarize_result, role=assistant/system, content=<report markdown>
step=submit_result, role=tool/system, content=<external submission summary>
```

The UI can group by `step` to preserve the graph presentation while using the
same underlying `messages` table as normal agent conversations.

## Live Streaming Semantics

There are two stream payload categories.

### Persisted Message Event

Persisted message events are durable and advance the client cursor:

```json
{
  "type": "message",
  "message": {
    "id": "uuid",
    "session_id": 1,
    "sequence": 12,
    "role": "assistant",
    "content": "...",
    "additional_kwargs": {
      "step": "review_sop",
      "kind": "final_message"
    },
    "created_at": "..."
  }
}
```

### Live Delta Event

Live token deltas are online-only and do not advance the cursor:

```json
{
  "type": "message_delta",
  "session_id": 1,
  "sequence": null,
  "role": "assistant",
  "content": "partial token",
  "additional_kwargs": {
    "step": "review_sop",
    "channel": "content"
  }
}
```

Thinking status can be a live-only signal:

```json
{
  "type": "message_delta",
  "session_id": 1,
  "sequence": null,
  "role": "assistant",
  "content": "",
  "additional_kwargs": {
    "step": "review_sop",
    "channel": "thinking",
    "status": "started"
  }
}
```

It must not include the model's private reasoning text.

## SSE Reconnection Semantics

The cursor is `messages.sequence`.

Flow:

1. Client fetches domain detail or session detail.
2. Client hydrates from persisted messages or a domain projection.
3. Client opens `GET /api/sessions/{session_id}/stream?after=<latest_sequence>`.
4. Server replays persisted `messages.sequence > after`.
5. Server subscribes to live broadcast.
6. Persisted message events advance the cursor.
7. Live delta events do not advance the cursor.
8. If the connection drops, reconnect with the last persisted sequence.
9. Missed token deltas are not replayed; the final persisted assistant message
   restores the complete content.

This gives deterministic replay without storing token-level noise.

## API Contract

### Generic Session APIs

Initial target APIs:

```http
GET /api/sessions/{session_id}
GET /api/sessions/{session_id}/messages?after=0
GET /api/sessions/{session_id}/stream?after=0
```

Optional lookup:

```http
GET /api/sessions/by-thread/{thread_id}
```

Session response:

```ts
type SessionDetail = {
  id: number;
  thread_id: string;
  status: "active" | "completed" | "failed" | "interrupted";
  title: string | null;
  latest_sequence: number;
  created_at: string;
  updated_at: string;
};
```

Message response:

```ts
type SessionMessage = {
  id: string;
  session_id: number;
  sequence: number;
  role: "user" | "assistant" | "tool" | "system";
  content: string;
  additional_kwargs: Record<string, unknown>;
  created_at: string;
};
```

### SOP API Integration

SOP detail should add:

```ts
type SopQualityCheckDetail = {
  check_id: string;
  session_id: number;
  thread_id: string;
  status: string;
  result: unknown;
  display_state: unknown;
};
```

First landing can keep:

```http
GET /api/sop-quality-checks/{check_id}/stream?after=0
```

Internally, it can map or proxy the generic session stream. Long term, frontend
can use:

```http
GET /api/sessions/{session_id}/stream?after=0
```

## Frontend Contract

Add a generic session hook:

```ts
useSessionStream(sessionId)
```

It owns:

- hydrate persisted messages
- open SSE
- reconnect with `after=latest_sequence`
- merge live deltas with persisted messages
- keep live delta buffers separate from durable transcript

SOP UI becomes a projection:

```ts
messages grouped by additional_kwargs.step
```

Suggested step display labels:

```ts
{
  load_sop: "Read SOP",
  review_sop: "DeepAgent Review",
  summarize_result: "Generate Summary",
  submit_result: "Submit Result"
}
```

The frontend should not display raw internal node names without a mapping. It
should not know DeepAgent chunk formats.

Agent playground UI can render the same messages as a conversation:

- `role=user`: user bubble
- `role=assistant`: assistant response
- `role=tool`: tool call/result card
- live `message_delta`: temporary assistant streaming buffer

## First Landing Scope

Keep the first implementation focused:

1. Add `sessions` and `messages` tables, models, repository, and schemas.
2. Add `sop_quality_checks.session_id`.
3. When starting a new SOP check, create a session first and use
   `sessions.thread_id` as the LangGraph thread ID.
4. Introduce `SessionMessageWriter`.
5. Introduce `DeepAgentStreamRunner`.
6. Move DeepAgent stream consumption out of `review_sop`.
7. Let ordinary SOP nodes append only user-visible step messages through the
   writer.
8. Persist final assistant/step messages to `messages`.
9. Broadcast token deltas live-only.
10. Keep existing SOP stream endpoint compatible during the transition.

Do not build the full agent playground UI in this first landing. Design the
foundation so it can use the same runtime next.

## Migration Strategy

1. Create `sessions` and `messages`.
2. Add nullable `sop_quality_checks.session_id`.
3. Backfill sessions for existing SOP checks if existing data must be preserved.
   For a development-only PR branch, this can be skipped or handled with a simple
   migration note.
4. Make `session_id` not null once new check creation always writes sessions.
5. Keep `sop_quality_checks.thread_id` during the transition.
6. Set `sop_quality_checks.thread_id = sessions.thread_id` for compatibility.
7. Move new transcript writes to `messages`.
8. Keep `sop_quality_events` read paths only until frontend and API no longer
   depend on them.

PostgreSQL 13 supports the required identity columns, JSONB defaults, foreign
keys, and unique indexes.

## Sequence Allocation

`messages.sequence` must be allocated under a session-level lock to avoid
duplicates:

```text
select sessions.id from sessions where id = :session_id for update
select coalesce(max(sequence), 0) + 1 from messages where session_id = :session_id
insert message with that sequence
```

Alternative implementations can store `latest_sequence` on `sessions`, but that
is not required for the first landing.

## Error Handling

- If DeepAgent streaming fails after partial live deltas, no partial assistant
  message is persisted.
- The business record and `sessions.status` are marked failed.
- A concise error can be stored in the business table error field.
- If a user reconnects after failure, they see persisted messages up to the last
  complete message and the failed business status.
- Startup interruption should mark active sessions and business rows
  interrupted when the corresponding run cannot resume automatically.

## Testing Strategy

Backend tests:

- `sessions/messages` models expose the expected columns and indexes.
- Repository allocates session-local `sequence` values monotonically.
- Repository enforces unique `(session_id, sequence)`.
- `SessionMessageWriter` writes persisted messages and broadcasts them.
- Live token deltas are not inserted into `messages`.
- `DeepAgentStreamRunner` uses `astream` when available.
- `DeepAgentStreamRunner` aggregates final content and writes one assistant
  message.
- `DeepAgentStreamRunner` emits thinking status without exposing reasoning text.
- SOP graph test proves `review_sop` uses `DeepAgentStreamRunner` and no longer
  contains stream parsing.
- SOP ordinary node tests prove they append step messages through the writer.
- SSE replay returns only persisted messages after the cursor.
- SSE live delta does not advance cursor.
- PostgreSQL database migration test covers `sessions`, `messages`, and
  `sop_quality_checks.session_id`.

Frontend tests:

- `useSessionStream` opens `after=<latest_sequence>`.
- Persisted message events advance the cursor.
- Live delta events do not advance the cursor.
- Reconnect after missing deltas restores final assistant content from persisted
  message.
- SOP projection groups messages by `additional_kwargs.step`.
- Tool messages render from `role=tool` and `additional_kwargs.kind/status`.

## Open Risks

- Commit frequency: writing a message before a long-running graph step gives
  better recovery, but too many commits can fragment graph transactions. The
  first landing should commit persisted transcript messages independently of
  live deltas.
- Existing SOP frontend code expects `sop_quality_events`. Compatibility mapping
  should be explicit and temporary.
- DeepAgent update chunks may expose tool events differently once MCP tools are
  configured. The first landing should support final messages and live deltas
  reliably, then add richer tool extraction when real tools are available.
- SOP `_user_message` may include full SOP snapshots. The transcript should
  store a product-facing user message and keep full SOP payloads in
  `sop_quality_checks.sop_snapshot` to avoid duplicating large content.
